# L1 — High-Level Requirements

## L1-1: Data Mining Service

The platform shall provide a .NET background service that discovers meetups and events from internet sources (Eventbrite, Meetup.com, and general web search) where Black professionals working in the space or defense industry are likely to attend or speak. Discovery shall also include .NET, Angular, and Playwright automated testing events. In-person events shall be scoped to the area near postal code L5A 4E6 (Mississauga, ON); online/virtual events shall be discovered regardless of location. Discovery shall be scoped to events occurring in the Toronto and Mississauga metropolitan area.

## L1-2: Data Store

The platform shall persist discovered events and their metadata to a relational data store (SQL Server via Entity Framework Core) so that data survives service restarts and can be queried efficiently.

## L1-3: Web API

The platform shall expose a RESTful ASP.NET Core Web API using Controllers that allows consumers to search, filter, and retrieve stored events and related metadata.

## L1-4: Solution Structure

The platform shall be organized as a clean .NET solution with clearly separated projects for the API, services, data access, and shared models.

## L1-5: UI Design

The platform shall have a modern Angular Material dark-mode UI that allows users to browse, search, and filter discovered events. All components, buttons, navigation, and typography shall use Angular Material and Material Design conventions. The design shall be created as a `.pen` file mockup covering the primary application screens.

## L1-6: Frontend Domain Models

The `Constellation.Web` domain library shall define TypeScript interfaces that mirror the backend DTOs, enabling type-safe communication between the Angular frontend and the ASP.NET Core API.

## L1-9: Frontend Intelligent Components

The `Constellation.Web` domain library shall provide intelligent (smart/container) Angular components corresponding to each screen in `ui-design.pen`. These components inject API services, manage state (loading, filtering, pagination), and compose presentational components from the `components` library. They include an App Shell with toolbar/sidenav/router, a Dashboard, an Events List, an Event Detail, and a Sources Overview.

## L1-7: Frontend API Services

The `Constellation.Web` api library shall provide injectable Angular services that encapsulate all HTTP communication with the Constellation API, returning typed observables for each endpoint.

## L1-8: Frontend Presentational Components

The `Constellation.Web` components library shall provide reusable, presentational Angular components matching the visual designs in `ui-design.pen`. Components shall be stateless (inputs/outputs only), use Angular Material dark-mode theming, and be independently usable across screens.

## L1-11: Playwright-Based Event Scraper

The platform shall include a Playwright-based scraper that uses browser automation to search real event websites (Eventbrite, Meetup.com, and general web search) for .NET, Angular, and Playwright events near postal code L5A 4E6 (Mississauga, ON) and online. The scraper shall extract actual event data from live web pages and seed the database with the discovered results. This scraper operates as a console tool that can be run on-demand, complementing the existing API-based event sources.

## L1-10: End-to-End Playwright Tests

The platform shall have a comprehensive Playwright E2E test suite using the Page Object Model pattern. Tests shall mock API responses, verify all screen workflows, and validate that rendered elements match the `ui-design.pen` design — including correct Angular Material icons, Roboto font usage, dark-mode colors, and spacing/alignment.

## L1-12: Responsive UI Design

The `ui-design.pen` file shall include responsive design mockups for mobile (402px width) and tablet (768px width) form factors covering all primary application screens (Dashboard, Events List, Event Detail, Sources Overview). Mobile layouts shall use a single-column stacked structure with a bottom tab bar for navigation. Tablet layouts shall use a compact two-column structure with a collapsed icon-rail sidenav. All responsive variants shall maintain the existing dark-mode Material Design visual language.

## L1-14: Responsive CSS Implementation

The Angular frontend shall implement responsive CSS so that the application layout adapts to mobile (≤480px), tablet (≤768px), and desktop (>768px) viewports, matching the responsive mockups in `ui-design.pen`. Mobile views shall use a bottom tab bar instead of the sidebar, tablet views shall use a collapsed icon-rail sidenav, and desktop views shall show the full sidebar. All components shall adapt their layout, spacing, and visibility to the active viewport.

## L1-15: Responsive Design Playwright Tests

The project shall include comprehensive Playwright E2E tests using the Page Object Model pattern that verify the responsive design implementation at mobile (402px), tablet (768px), and desktop (1440px) viewport widths. Tests shall validate layout changes, element visibility, navigation patterns, and component adaptations across all breakpoints.

## L1-13: Notifications, Errors, and Dialogs

The `ui-design.pen` file shall include designs for all system feedback UI: toast notifications (success, error, warning, info), error/empty states (network error, 404 not found, empty results), and modal dialogs (delete confirmation, filter panel, share event). These shall be designed as reusable component frames following Material Design patterns and the existing dark-mode theme.
