# L2 — Detailed Requirements

---

## L1-1: Data Mining Service

### L2-1.1: Event Source Abstraction

The service shall define an `IEventSource` interface so new data sources can be added without modifying existing code.

**Acceptance Criteria:**
- An `IEventSource` interface exists with a method that returns discovered events.
- At least three concrete implementations exist (Eventbrite, Meetup, WebSearch).
- A unit test verifies each source can be instantiated and implements the interface.

### L2-1.2: Eventbrite Source

An `EventbriteEventSource` shall query the Eventbrite API for events matching keywords related to Black professionals in space/defense (e.g., "Black in aerospace", "BEYA STEM", "National Society of Black Engineers defense", "Blacks in Government space").

**Acceptance Criteria:**
- The source sends HTTP requests to the Eventbrite API with relevant search terms.
- Queries are scoped to the Toronto/Mississauga area using location parameters (latitude/longitude or city name).
- Returned events are mapped to the shared `DiscoveredEvent` model.
- A test verifies the mapping logic with a stubbed HTTP response.

### L2-1.3: Meetup Source

A `MeetupEventSource` shall query the Meetup.com GraphQL API for groups and events matching relevant keywords.

**Acceptance Criteria:**
- The source sends GraphQL queries to the Meetup API.
- Queries are scoped to the Toronto/Mississauga area using location parameters.
- Returned events are mapped to the shared `DiscoveredEvent` model.
- A test verifies the mapping logic with a stubbed HTTP response.

### L2-1.4: Web Search Source

A `WebSearchEventSource` shall use a web search API (Bing Search API) to find event pages matching relevant queries.

**Acceptance Criteria:**
- The source sends HTTP requests to the Bing Search API with relevant search terms.
- Search queries include "Toronto" or "Mississauga" as geographic qualifiers.
- Returned results are mapped to the shared `DiscoveredEvent` model.
- A test verifies the mapping logic with a stubbed HTTP response.

### L2-1.5: Geographic Configuration

All event sources shall use a shared geographic configuration that defines the target area as Toronto and Mississauga, Ontario, Canada. The configuration shall include city names, latitude/longitude coordinates (Toronto: 43.6532, -79.3832; Mississauga: 43.5890, -79.6441), and a search radius (default: 50 km).

**Acceptance Criteria:**
- A `MiningOptions` configuration class exists with geographic properties (Cities, Latitude, Longitude, RadiusKm).
- Default values target the Toronto/Mississauga area.
- Configuration is bindable from `appsettings.json` under a `Mining` section.
- A test verifies default configuration values.

### L2-1.6: Mining Orchestrator Background Service

A `MiningBackgroundService` (hosted service) shall periodically invoke all registered `IEventSource` implementations with the configured geographic parameters, deduplicate results, and persist them via the data access layer.

**Acceptance Criteria:**
- The service runs on a configurable interval (default: every 6 hours).
- Duplicate events (same URL or name+date) are not re-inserted.
- A test verifies deduplication logic.

## L1-2: Data Store

### L2-2.1: Entity Model

The data layer shall define an `Event` entity with properties: Id, Title, Description, Url, Location, StartDate, EndDate, Source, Tags, RelevanceScore, CreatedAt, UpdatedAt.

**Acceptance Criteria:**
- The `Event` entity class exists with all listed properties.
- A unit test verifies all properties are present.

### L2-2.2: EF Core DbContext

A `ConstellationDbContext` shall be configured with an `Events` DbSet and use SQL Server as the provider.

**Acceptance Criteria:**
- The DbContext class exists and configures the Events table.
- An initial EF Core migration can be generated successfully.
- A test verifies the DbContext can be instantiated with an in-memory provider.

### L2-2.3: Event Repository

An `IEventRepository` / `EventRepository` pair shall encapsulate data access operations: Add, GetById, GetAll (with paging), Search (by keyword, date range, source, tags).

**Acceptance Criteria:**
- CRUD and search methods are implemented.
- A test verifies add and search operations using the in-memory database provider.

## L1-3: Web API

### L2-3.1: Events Controller

An `EventsController` shall expose the following endpoints:
- `GET /api/events` — list events with paging, filtering by keyword, date range, source, and tags.
- `GET /api/events/{id}` — get a single event by ID.

**Acceptance Criteria:**
- Both endpoints return correct HTTP status codes (200, 404).
- Paging defaults to page 1, size 20.
- A test verifies each endpoint using `WebApplicationFactory`.

### L2-3.2: Search Endpoint

An `EventsController` shall expose:
- `GET /api/events/search?q={query}` — full-text keyword search across Title, Description, Tags.

**Acceptance Criteria:**
- Returns matching events with 200 OK.
- Returns empty array with 200 OK when no matches.
- A test verifies search behavior using `WebApplicationFactory`.

### L2-3.3: Sources Controller

A `SourcesController` shall expose:
- `GET /api/sources` — list all distinct event sources with event counts.

**Acceptance Criteria:**
- Returns a list of source names with their event counts.
- A test verifies the endpoint using `WebApplicationFactory`.

### L2-3.4: Health Check Endpoint

The API shall expose a `GET /health` endpoint.

**Acceptance Criteria:**
- Returns 200 OK with a JSON body containing status and timestamp.
- A test verifies the health endpoint.

## L1-4: Solution Structure

### L2-4.1: Solution Layout

The solution shall be organized as:
```
Constellation.sln
src/
  Constellation.Api/          — ASP.NET Core Web API (Controllers, Program.cs)
  Constellation.Services/     — Data mining services and background workers
  Constellation.Data/         — EF Core DbContext, migrations, repositories
  Constellation.Models/       — Shared DTOs and entity models
tests/
  Constellation.Tests/        — Unit and integration tests
```

**Acceptance Criteria:**
- The solution file references all five projects.
- Each project builds successfully.
- `dotnet build` succeeds at the solution level.

### L2-4.2: Dependency Injection Configuration

The API project's `Program.cs` shall register all services (repositories, event sources, background service, DbContext) via dependency injection.

**Acceptance Criteria:**
- All services are registered in the DI container.
- The application starts without DI resolution errors.
- A test verifies the application can be built via `WebApplicationFactory`.

## L1-5: UI Design

### L2-5.1: App Shell and Navigation

The UI shall have an Angular Material dark-mode app shell consisting of a `mat-toolbar` top app bar with the Constellation brand name and a `mat-sidenav` side navigation with links to Dashboard, Events, Sources, and Settings. Typography shall use Material Design type scale.

**Acceptance Criteria:**
- A dark-themed top toolbar displays the "Constellation" brand name with a menu toggle icon button.
- A side navigation drawer contains nav items for Dashboard, Events, Sources, and Settings, each with a Material icon.
- The layout uses proper Material dark-mode surface colors (#121212 background, #1E1E1E surfaces).
- All text uses the Material Design type scale (Roboto font family).

### L2-5.2: Dashboard Screen

The UI shall have a Dashboard screen showing summary stat cards (Total Events, Sources Active, Events This Week, Avg Relevance Score) using `mat-card` components, followed by a "Recent Events" section with a `mat-table` showing the latest events.

**Acceptance Criteria:**
- Four `mat-card` stat cards are displayed in a responsive grid row, each with a Material icon, metric value, and label.
- A "Recent Events" `mat-table` displays columns: Title, Date, Source, Relevance with sample data rows.
- Cards use Material elevation and dark surface colors.
- A `mat-raised-button` "View All Events" appears below the table.

### L2-5.3: Events List Screen

The UI shall have an Events screen with a search/filter bar and a paginated event list. The filter bar uses `mat-form-field` with `mat-input` for keyword search, `mat-select` for source filter, and `mat-datepicker` for date range. Events are displayed as `mat-card` items showing title, date, source chip, relevance badge, and description excerpt.

**Acceptance Criteria:**
- A filter bar contains a search input, source dropdown, and date range picker, all using Angular Material form field components.
- Event cards display title, date, source as a `mat-chip`, relevance as a colored badge, and a truncated description.
- A `mat-paginator` appears at the bottom of the event list.
- All form controls use Material dark-mode theming (filled appearance).

### L2-5.4: Event Detail View

The UI shall have an Event Detail view displayed as a full-width `mat-card` with the event title, full description, metadata (date, location, source, relevance score), tags as `mat-chip-set`, and action buttons (Open Link, Share) using `mat-raised-button` and `mat-stroked-button`.

**Acceptance Criteria:**
- The detail card displays the event title as a `mat-card-title` with headline typography.
- Metadata fields (Date, Location, Source, Relevance) are listed with Material icons.
- Tags are displayed as a `mat-chip-set` with individual `mat-chip` components.
- Two action buttons appear: "Open Link" (raised, accent color) and "Share" (stroked).

### L2-5.5: Sources Overview Screen

The UI shall have a Sources screen showing each data source as a `mat-card` with the source name, event count, last sync timestamp, and a status indicator. A `mat-progress-bar` shows the relative proportion of events from each source.

**Acceptance Criteria:**
- Each source is displayed as a `mat-card` with source name, event count, last synced time, and a status chip (Active/Inactive).
- A `mat-progress-bar` inside each card visualizes the event count relative to total.
- Cards are arranged in a grid layout with proper spacing.
- Uses Material dark-mode card styling.

## L1-6: Frontend Domain Models

### L2-6.1: Event Model Interface

The domain library shall export a TypeScript `EventDto` interface matching the backend DTO with properties: id, title, description, url, location, startDate, endDate, source, tags (string array), relevanceScore.

**Acceptance Criteria:**
- An `EventDto` interface exists in `projects/domain/src/lib/models/event.model.ts`.
- All properties match the backend `EventDto` C# class (using camelCase).
- The interface is exported from the library's public API.
- A unit test verifies the interface can be used to type an object.

### L2-6.2: Search and Paging Model Interfaces

The domain library shall export `EventSearchRequest`, `PagedResult<T>`, and `SourceSummaryDto` interfaces matching their backend counterparts.

**Acceptance Criteria:**
- `EventSearchRequest` interface has optional keyword, fromDate, toDate, source, tag, and required page/pageSize.
- `PagedResult<T>` interface has items (T[]), totalCount, page, pageSize, totalPages.
- `SourceSummaryDto` interface has source and eventCount.
- All interfaces are exported from the library's public API.
- A unit test verifies each interface can be used for typing.

## L1-7: Frontend API Services

### L2-7.1: Events API Service

The api library shall provide an `EventsService` injectable service with methods: `getEvents(params)`, `getEvent(id)`, and `searchEvents(query)` that call the corresponding backend endpoints and return typed observables.

**Acceptance Criteria:**
- `EventsService` is an `@Injectable({ providedIn: 'root' })` service.
- `getEvents` calls `GET /api/events` with query parameters and returns `Observable<PagedResult<EventDto>>`.
- `getEvent` calls `GET /api/events/{id}` and returns `Observable<EventDto>`.
- `searchEvents` calls `GET /api/events/search?q={query}` and returns `Observable<EventDto[]>`.
- A unit test verifies each method constructs the correct HTTP request using `HttpClientTestingModule`.

### L2-7.2: Sources API Service

The api library shall provide a `SourcesService` injectable service with a `getSources()` method that calls `GET /api/sources` and returns typed observables.

**Acceptance Criteria:**
- `SourcesService` is an `@Injectable({ providedIn: 'root' })` service.
- `getSources` calls `GET /api/sources` and returns `Observable<SourceSummaryDto[]>`.
- A unit test verifies the method constructs the correct HTTP request.

## L1-8: Frontend Presentational Components

### L2-8.1: Sidenav Component

A `SidenavComponent` shall render a vertical navigation sidebar with the Constellation brand logo/text and nav items (Dashboard, Events, Sources, Settings), each with a Material icon. The active item shall be visually highlighted.

**Acceptance Criteria:**
- Component accepts `activeRoute` input to highlight the current nav item.
- Component emits `navItemClicked` output with the route string when a nav item is clicked.
- Renders brand icon (stars) and "Constellation" text at top.
- Four nav items with icons: dashboard, event, cloud_sync, settings.
- Active item has highlighted background (#2C2C2C) and accent-colored icon (#BB86FC).
- Uses dark surface color (#1A1A1A) background.
- A unit test verifies the component renders and emits navigation events.

### L2-8.2: Stat Card Component

A `StatCardComponent` shall render a summary statistic card with an icon, metric value, label, and trend indicator.

**Acceptance Criteria:**
- Component accepts inputs: `icon` (string), `value` (string), `label` (string), `trend` (string), `trendUp` (boolean).
- Renders a Material icon in a circular container.
- Displays the metric value in large text and label below.
- Shows trend with a trending_up/trending_down icon and colored text (green for up, red for down).
- Uses dark card styling (#1E1E1E background, 12px corner radius, 24px padding).
- A unit test verifies the component renders all inputs correctly.

### L2-8.3: Toolbar Component

A `ToolbarComponent` shall render the top app bar with a menu toggle button, the "Constellation" brand title, and action icons (search, notifications, avatar).

**Acceptance Criteria:**
- Component emits `menuToggled` output when menu icon is clicked.
- Renders menu icon, "Constellation" title, search icon, notification icon, and avatar circle.
- Uses dark toolbar color (#1F1F1F) with bottom border.
- A unit test verifies the component renders and emits the toggle event.

### L2-8.4: Event Card Component

An `EventCardComponent` shall render an event list item card showing the event title, relevance badge, metadata row (date, location, source chip), and description excerpt.

**Acceptance Criteria:**
- Component accepts an `event` input of type `EventDto`.
- Component emits `eventClicked` output with the event when clicked.
- Displays title and relevance score badge in the header row.
- Shows date with calendar icon, location with pin icon, and source as a chip.
- Truncated description text below metadata.
- Uses dark card styling (#1E1E1E, 12px radius, 24px padding).
- A unit test verifies the component renders event data and emits click events.

### L2-8.5: Source Card Component

A `SourceCardComponent` shall render a data source card with source name, status chip, event count, last sync time, and a progress bar showing event share.

**Acceptance Criteria:**
- Component accepts inputs: `name`, `icon`, `iconColor`, `eventCount`, `lastSync`, `status`, `sharePercent`.
- Displays source icon in a circle, name, and Active/Inactive status chip.
- Shows event count and last sync as stat columns.
- Progress bar with colored fill representing the share percentage.
- Uses dark card styling (#1E1E1E, 12px radius, 24px padding).
- A unit test verifies the component renders all data correctly.

### L2-8.6: Event Detail Card Component

An `EventDetailCardComponent` shall render a full event detail view with title, relevance badge, metadata grid, description, tags, and action buttons.

**Acceptance Criteria:**
- Component accepts an `event` input of type `EventDto`.
- Component emits `openLink` and `share` outputs.
- Displays event title with star-rated relevance badge.
- Metadata grid shows date, location, source, and relevance score with Material icons.
- Full description text section.
- Tags rendered as chip pills.
- "Share" (stroked) and "Open Link" (filled teal) action buttons.
- A unit test verifies the component renders all sections and emits button events.

### L2-8.7: Data Table Component

A `DataTableComponent` shall render a tabular view of events with configurable columns and row data.

**Acceptance Criteria:**
- Component accepts `columns` input (array of column definitions) and `rows` input (EventDto array).
- Component emits `rowClicked` output with the clicked event.
- Renders header row with column labels in uppercase.
- Data rows show title, date, source chip, and relevance badge.
- Alternating row dividers for visual separation.
- Uses dark surface (#1E1E1E) with header background (#2C2C2C).
- A unit test verifies the component renders columns and rows.

### L2-8.8: Paginator Component

A `PaginatorComponent` shall render pagination controls with items-per-page selector, page info text, and previous/next buttons.

**Acceptance Criteria:**
- Component accepts inputs: `totalItems`, `pageSize`, `currentPage`.
- Component emits `pageChanged` output with the new page number and `pageSizeChanged` with the new size.
- Displays "Items per page:" with a selector, current range text, and prev/next arrow buttons.
- Disables prev button on first page and next button on last page.
- A unit test verifies pagination logic and output emissions.
