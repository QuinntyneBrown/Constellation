# L2 — Detailed Requirements

---

## L1-1: Data Mining Service

### L2-1.1: Event Source Abstraction

The service shall define an `IEventSource` interface so new data sources can be added without modifying existing code.

**Acceptance Criteria:**
- An `IEventSource` interface exists with a method that returns discovered events.
- At least three concrete implementations exist (Eventbrite, Meetup, WebSearch).
- A unit test verifies each source can be instantiated and implements the interface.

### L2-1.2: Eventbrite Source

An `EventbriteEventSource` shall query the Eventbrite API for events matching keywords related to Black professionals in space/defense (e.g., "Black in aerospace", "BEYA STEM", "National Society of Black Engineers defense", "Blacks in Government space").

**Acceptance Criteria:**
- The source sends HTTP requests to the Eventbrite API with relevant search terms.
- Returned events are mapped to the shared `DiscoveredEvent` model.
- A test verifies the mapping logic with a stubbed HTTP response.

### L2-1.3: Meetup Source

A `MeetupEventSource` shall query the Meetup.com GraphQL API for groups and events matching relevant keywords.

**Acceptance Criteria:**
- The source sends GraphQL queries to the Meetup API.
- Returned events are mapped to the shared `DiscoveredEvent` model.
- A test verifies the mapping logic with a stubbed HTTP response.

### L2-1.4: Web Search Source

A `WebSearchEventSource` shall use a web search API (Bing Search API) to find event pages matching relevant queries.

**Acceptance Criteria:**
- The source sends HTTP requests to the Bing Search API with relevant search terms.
- Returned results are mapped to the shared `DiscoveredEvent` model.
- A test verifies the mapping logic with a stubbed HTTP response.

### L2-1.5: Mining Orchestrator Background Service

A `MiningBackgroundService` (hosted service) shall periodically invoke all registered `IEventSource` implementations, deduplicate results, and persist them via the data access layer.

**Acceptance Criteria:**
- The service runs on a configurable interval (default: every 6 hours).
- Duplicate events (same URL or name+date) are not re-inserted.
- A test verifies deduplication logic.

## L1-2: Data Store

### L2-2.1: Entity Model

The data layer shall define an `Event` entity with properties: Id, Title, Description, Url, Location, StartDate, EndDate, Source, Tags, RelevanceScore, CreatedAt, UpdatedAt.

**Acceptance Criteria:**
- The `Event` entity class exists with all listed properties.
- A unit test verifies all properties are present.

### L2-2.2: EF Core DbContext

A `ConstellationDbContext` shall be configured with an `Events` DbSet and use SQL Server as the provider.

**Acceptance Criteria:**
- The DbContext class exists and configures the Events table.
- An initial EF Core migration can be generated successfully.
- A test verifies the DbContext can be instantiated with an in-memory provider.

### L2-2.3: Event Repository

An `IEventRepository` / `EventRepository` pair shall encapsulate data access operations: Add, GetById, GetAll (with paging), Search (by keyword, date range, source, tags).

**Acceptance Criteria:**
- CRUD and search methods are implemented.
- A test verifies add and search operations using the in-memory database provider.

## L1-3: Web API

### L2-3.1: Events Controller

An `EventsController` shall expose the following endpoints:
- `GET /api/events` — list events with paging, filtering by keyword, date range, source, and tags.
- `GET /api/events/{id}` — get a single event by ID.

**Acceptance Criteria:**
- Both endpoints return correct HTTP status codes (200, 404).
- Paging defaults to page 1, size 20.
- A test verifies each endpoint using `WebApplicationFactory`.

### L2-3.2: Search Endpoint

An `EventsController` shall expose:
- `GET /api/events/search?q={query}` — full-text keyword search across Title, Description, Tags.

**Acceptance Criteria:**
- Returns matching events with 200 OK.
- Returns empty array with 200 OK when no matches.
- A test verifies search behavior using `WebApplicationFactory`.

### L2-3.3: Sources Controller

A `SourcesController` shall expose:
- `GET /api/sources` — list all distinct event sources with event counts.

**Acceptance Criteria:**
- Returns a list of source names with their event counts.
- A test verifies the endpoint using `WebApplicationFactory`.

### L2-3.4: Health Check Endpoint

The API shall expose a `GET /health` endpoint.

**Acceptance Criteria:**
- Returns 200 OK with a JSON body containing status and timestamp.
- A test verifies the health endpoint.

## L1-4: Solution Structure

### L2-4.1: Solution Layout

The solution shall be organized as:
```
Constellation.sln
src/
  Constellation.Api/          — ASP.NET Core Web API (Controllers, Program.cs)
  Constellation.Services/     — Data mining services and background workers
  Constellation.Data/         — EF Core DbContext, migrations, repositories
  Constellation.Models/       — Shared DTOs and entity models
tests/
  Constellation.Tests/        — Unit and integration tests
```

**Acceptance Criteria:**
- The solution file references all five projects.
- Each project builds successfully.
- `dotnet build` succeeds at the solution level.

### L2-4.2: Dependency Injection Configuration

The API project's `Program.cs` shall register all services (repositories, event sources, background service, DbContext) via dependency injection.

**Acceptance Criteria:**
- All services are registered in the DI container.
- The application starts without DI resolution errors.
- A test verifies the application can be built via `WebApplicationFactory`.
